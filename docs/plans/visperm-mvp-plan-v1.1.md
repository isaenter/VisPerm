## 一、RALPLAN-DR 摘要

### 1.1 设计原则 (Principles)

| 编号 | 原则 | 说明 | 优先级 |
|------|------|------|--------|
| **P1** | 权限计算结果必须 100% 准确 | 宁可拒绝合法请求也不授予非法权限 | P0 |
| **P2** | 异步预计算 + 实时缓存 | 离线/定时预计算权限快照，查询时直接从缓存返回；缓存 Miss 时降级为实时图遍历计算 | P0 |
| **P3** | 拓扑变更触发增量重算 | 保证缓存与画布状态最终一致 | P1 |
| **P4** | 支持降级模式 | 当缓存不可用时，降级为实时图遍历计算 | P1 |
| **P5** | 所有权限计算必须记录审计日志 | 支持追溯和回放 | P2 |

### 1.2 决策驱动因素 (Decision Drivers)

| 优先级 | 驱动因素 | 影响范围 | 应对策略 |
|--------|----------|----------|----------|
| **DD1** | ERP/MES 企业级场景复杂度 | 需要支持大规模节点 (1000+) 和复杂拓扑关系 | 分层缓存 + 增量重算 |
| **DD2** | 非技术人员易用性要求 | 界面必须直观，拖拽操作零学习成本 | LogicFlow 画布 + 自定义节点 |
| **DD3** | 高可用性和容错要求 | 必须具备缓存降级、异常熔断能力 | Redis 缓存 + 降级策略 |

### 1.3 可行方案对比 (Viable Options)

#### 方案 A：纯关系型数据库方案 (PostgreSQL + Prisma + Redis 缓存)
| 优点 | 缺点 |
|------|------|
| ✅ 技术栈统一，团队学习成本低 | ❌ 深层递归查询性能受限 |
| ✅ 事务支持完善，数据一致性强 | ❌ 图算法需要应用层实现 |
| ✅ Prisma ORM 开发效率高 | ❌ 复杂拓扑验证逻辑复杂 |

#### 方案 B：混合存储方案 (PostgreSQL + Neo4j)
| 优点 | 缺点 |
|------|------|
| ✅ 原生图存储，遍历性能优异 | ❌ 图数据库运维成本高 |
| ✅ Cypher 查询语言简洁 | ❌ 双写一致性难保证 |
| ✅ 适合超大规模拓扑 (10000+ 节点) | ❌ MVP 阶段过度设计 |

#### 方案 C：Redis 图缓存方案 (PostgreSQL + RedisGraph)
| 优点 | 缺点 |
|------|------|
| ✅ 内存计算，响应速度极快 | ❌ Redis 持久化可靠性弱于 PG |
| ✅ 天然支持缓存降级策略 | ❌ 内存成本较高 |
| ✅ 与现有技术栈 (Redis) 无缝集成 | ❌ 需要额外数据同步逻辑 |

**MVP 阶段推荐方案：方案 A**
- 核心数据存 PostgreSQL，保证事务一致性
- 热点拓扑数据缓存在 Redis，支持快速遍历
- 降级时直接返回缓存权限快照

**排除方案 B 的理由**：Neo4j 引入运维复杂度，MVP 阶段收益不明确，可在规模扩展时再评估。

## 三、Pre-mortem 分析（3 场景）

### 场景 1：权限计算性能在生产环境不达标

| 项目 | 内容 |
|------|------|
| **假设失败** | MVP 上线后，权限计算 P95 延迟超过 500ms（目标<100ms） |
| **概率** | 中 (40%) |
| **影响** | 高 (用户等待超时，系统不可用) |
| **根本原因** | 1. 5000+ 节点全图 DFS 遍历耗时过长<br>2. Prisma N+1 查询问题未优化<br>3. 缓存命中率低于 50% |
| **预防措施** | 1. W5 增加性能基准测试<br>2. 提前实现物化视图方案<br>3. 引入缓存预热机制<br>4. 数据库查询添加索引 |
| **应急方案** | 1. 降级为预计算快照模式<br>2. 限制单次计算最大节点数<br>3. 异步计算 + 轮询结果 |
| **触发条件** | P95 延迟 >300ms 持续 5 分钟 |
| **监控指标** | `perm_calculation_duration_seconds`, `perm_cache_hit_total` |

### 场景 2：LogicFlow 画布无法承载 1000+ 节点渲染

| 项目 | 内容 |
|------|------|
| **假设失败** | 前端画布在 500 节点时已出现明显卡顿，无法达成"1000+ 节点秒级加载"目标 |
| **概率** | 中 (35%) |
| **影响** | 高 (用户无法编辑大型拓扑，产品价值受损) |
| **根本原因** | 1. LogicFlow Canvas 渲染性能瓶颈<br>2. 未实现虚拟滚动和分层渲染<br>3. 节点组件过于复杂 |
| **预防措施** | 1. W2 进行技术预研验证<br>2. 预留 AntV X6 迁移接口<br>3. 简化默认节点样式<br>4. 实现节点聚合显示 |
| **应急方案** | 1. 切换至 AntV X6<br>2. 启用节点聚合显示<br>3. 限制单画布最大节点数 |
| **触发条件** | 500 节点时帧率 <20 FPS |
| **监控指标** | `canvas_render_time_ms`, `canvas_fps` |

### 场景 3：缓存与降级机制引入数据一致性问题

| 项目 | 内容 |
|------|------|
| **假设失败** | 用户反馈权限变更后未实时生效，或返回过期权限数据 |
| **概率** | 高 (60%) |
| **影响** | 高 (权限泄露或授权失败，安全事故) |
| **根本原因** | 1. 缓存失效机制有漏洞<br>2. 降级逻辑返回了陈旧数据<br>3. 事务边界不清晰 |
| **预防措施** | 1. 实现拓扑版本号机制<br>2. 降级时明确标记数据新鲜度<br>3. 写操作强制刷新缓存<br>4. 定期一致性校验 |
| **应急方案** | 1. 一键清空缓存<br>2. 降级开关可配置<br>3. 审计日志追踪权限来源 |
| **触发条件** | 收到权限不一致报告 |
| **监控指标** | `cache_consistency_check_fail_total`, `perm_degrade_mode_active` |

## 五、原则 - 方案一致性说明

### P2 原则修订说明

**原表述**：
> P2: 实时计算权限，响应时间 < 100ms

**修订后表述**：
> P2: 采用"异步预计算 + 实时缓存"策略：离线/定时预计算权限快照，查询时直接从缓存返回；缓存 Miss 时降级为实时图遍历计算

**语义冲突协调**：

| 原语义 | 冲突点 | 修订后语义 |
|--------|--------|------------|
| "实时计算"暗示每次请求都遍历图 | 与性能目标冲突 | 明确"预计算为主，实时计算为降级兜底" |
| 未说明缓存策略 | 无法应对高并发 | 增加 L1/L2 分层缓存设计 |
| 未说明降级机制 | 单点故障风险 | 明确降级路径和触发条件 |

**架构一致性图**：

```
┌─────────────────────────────────────────────────────────────┐
│                    权限计算架构                              │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐   │
│  │   用户请求   │ ──→ │   缓存层    │ ──→ │   返回结果   │   │
│  └─────────────┘     └──────┬──────┘     └─────────────┘   │
│                             │ Miss                          │
│                             ↓                               │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐   │
│  │  预计算任务  │ ←─┐ │  图遍历引擎  │ ──→ │  写入缓存   │   │
│  └─────────────┘   │ └─────────────┘     └─────────────┘   │
│                    │                                        │
│  ┌─────────────┐   │                                        │
│  │  拓扑变更事件│ ──┘ (触发增量重算)                         │
│  └─────────────┘                                           │
└─────────────────────────────────────────────────────────────┘
```

# ADR-001: VisPerm MVP 技术选型与架构决策

## 文档信息

| 字段 | 值 |
|------|-----|
| ADR 编号 | ADR-001 |
| 创建日期 | 2026/04/09 |
| 状态 | 已批准 |

---

## 决策概述 (Decision)

**决策**：VisPerm MVP 采用"PostgreSQL + Prisma + Redis 缓存"技术栈，20 周实施计划，分 6 个 Phase 交付。

---

## 决策驱动因素 (Drivers)

| 优先级 | 驱动因素 | 说明 |
|--------|----------|------|
| DD1 | ERP/MES 企业级场景复杂度 | 需要支持大规模节点 (1000+) 和复杂拓扑关系 |
| DD2 | 非技术人员易用性要求 | 界面必须直观，拖拽操作零学习成本 |
| DD3 | 高可用性和容错要求 | 必须具备缓存降级、异常熔断能力 |

---

## 考虑的替代方案 (Alternatives Considered)

### 方案 A：纯关系型数据库方案 (PostgreSQL + Prisma + Redis 缓存) ✅ 已选
| 优点 | 缺点 |
|------|------|
| ✅ 技术栈统一，团队学习成本低 | ❌ 深层递归查询性能受限 |
| ✅ 事务支持完善，数据一致性强 | ❌ 图算法需要应用层实现 |
| ✅ Prisma ORM 开发效率高 | ❌ 复杂拓扑验证逻辑复杂 |

### 方案 B：混合存储方案 (PostgreSQL + Neo4j)
| 优点 | 缺点 |
|------|------|
| ✅ 原生图存储，遍历性能优异 | ❌ 图数据库运维成本高 |
| ✅ Cypher 查询语言简洁 | ❌ 双写一致性难保证 |
| ✅ 适合超大规模拓扑 (10000+ 节点) | ❌ MVP 阶段过度设计 |

**排除理由**：
- Neo4j 引入运维复杂度，与团队现有技术栈差异大
- MVP 阶段节点规模预期 <5000，PostgreSQL 递归查询可满足
- 可在规模扩展时再评估迁移

### 方案 C：Redis 图缓存方案 (PostgreSQL + RedisGraph)
| 优点 | 缺点 |
|------|------|
| ✅ 内存计算，响应速度极快 | ❌ Redis 持久化可靠性弱于 PG |
| ✅ 天然支持缓存降级策略 | ❌ 内存成本较高 |
| ✅ 与现有技术栈 (Redis) 无缝集成 | ❌ 需要额外数据同步逻辑 |

**部分采纳**：Redis 缓存策略已纳入方案 A，但使用标准 Redis 数据结构而非 RedisGraph。

---

## 选择理由 (Why Chosen)

选择方案 A 的核心理由：

1. **风险最小化**：技术栈成熟，团队熟悉度高，MVP 交付风险最低
2. **一致性保证**：PostgreSQL 事务支持完善，权限数据一致性有保证
3. **渐进式演进**：可在后期根据实际性能瓶颈，选择引入 Neo4j 或优化现有方案
4. **运维友好**：无需引入新组件，降低运维复杂度

---

## 后果与权衡 (Consequences)

### 正面后果
- 开发效率高，团队可快速上手
- 数据一致性强，权限计算结果可靠
- 运维成本低，基础设施复用现有资源

### 负面后果
- 需要自行实现图遍历算法，开发复杂度增加
- 大规模拓扑（5000+ 节点）性能需持续优化
- 需要额外的性能压测和优化周期（Phase 5）

### 技术债务
- 图遍历算法性能优化空间有限，可能需要在规模扩展时重构
- 缓存一致性机制需要持续维护
